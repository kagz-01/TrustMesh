PROJECT 01
TrustMesh
Decentralized AI-Driven Zero Trust Security Platform
	

Overview & Strategic Context
TrustMesh is a cloud-native security platform that eliminates implicit trust in microservice environments. It enforces continuous, multi-factor verification across identity, behaviour, and policy for every request, internal or external. This approach aligns with the 2025 industry consensus that Zero Trust has evolved from a framework into an operational necessity.
According to a 2023 Gartner survey, 63% of organisations globally had fully or partially implemented a Zero Trust strategy. By 2025, leading vendors including Palo Alto Networks, Zscaler, CrowdStrike, and cloud-native innovators like AccuKnox and Prophaze have standardised on Zero Trust as the baseline for enterprise security. The shift is driven by the realisation that cloud-native environments — serverless functions, ephemeral containers, distributed microservices — break the assumptions of perimeter-based models.


Problem Space
Traditional cloud systems grant persistent trust after a single authentication event. This creates three classes of risk:
✔  Lateral Movement Attacks — attackers who breach one service escalate freely across the network
	✔  Compromised Service Accounts — static credentials become persistent backdoors
	✔  Insider Threats — authorised identities abused by malicious or negligent actors
	✔  AI-Augmented Attacks — adversaries using machine-speed automation for social engineering and credential stuffing
	✔  Dynamic Workload Risk — serverless and ephemeral containers defeat static policy enforcement
	

How It Works — Detailed Flow
TrustMesh operates as a policy-enforcement overlay on top of a Kubernetes service mesh. Every network interaction is intercepted and evaluated before being allowed.
1
	Service or user initiates a request — the API Gateway intercepts before any payload reaches the destination service.
	2
	Blockchain Identity Verification — the requesting entity's identity is resolved against a decentralised identity registry stored on-chain. Only cryptographically verified identities proceed.
	3
	Behavioural Risk Scoring — the AI engine runs a real-time anomaly score against the request's metadata: timing, volume, endpoint pattern, geolocation, and deviation from baseline.
	4
	Policy Evaluation — the Open Policy Agent (OPA) evaluates the request against active Rego policies, considering identity, risk score, resource sensitivity, and time.
	5
	Allow / Deny — permitted requests are forwarded; denied requests are blocked and trigger an alert. Borderline requests can trigger step-up authentication.
	6
	Immutable Audit Log — every decision (allow or deny) is written to IPFS, creating a tamper-evident audit trail. Prometheus scrapes event metrics; Grafana renders the live dashboard.

PROJECT 02
OpsShield
Autonomous DevSecOps Security Platform
	

Overview & Strategic Context
PipelineSentinel is a fully automated DevSecOps platform that embeds security controls into every stage of the CI/CD pipeline — from first commit to production deployment. The 2025 DevSecOps landscape is defined by the principle of 'shift-left security': vulnerabilities found at design time cost 100x less to fix than those discovered post-release.
Industry research from Datadog's State of DevSecOps 2025 confirms that only 18% of identified vulnerabilities are truly critical, yet most teams lack tools to distinguish them. AI-powered prioritisation is now the differentiator. CrowdStrike's March 2025 addition of AI-powered Network Vulnerability Assessment within Falcon Exposure Management highlights how major vendors are embedding ML-driven triage directly into DevSecOps workflows.


Problem Space
✔  Dependency Attacks — malicious or outdated packages silently introduced via npm, pip, Maven
	✔  Build Artifact Tampering — unsigned binaries swapped during CI/CD transit
	✔  Secret Leaks — API keys, tokens, passwords committed to repositories
	✔  Container Vulnerabilities — base image CVEs reaching production undetected
	✔  Infrastructure Drift — IaC misconfigurations creating exploitable cloud resources
	✔  Supply Chain Compromise — third-party CI actions or libraries injecting malicious code
	

How It Works — Pipeline Flow
1
	Developer commits code — pre-commit hooks run SAST (static analysis) and secret detection immediately.
	2
	Pull Request opened — AI engine scans changed files, scores risk, flags high-severity issues inline in the PR.
	3
	Build Stage — SCA (software composition analysis) checks all dependencies against CVE databases. Trivy scans the container image. Only images passing vulnerability thresholds proceed.
	4
	Artifact Signing — cosign signs the container image via OIDC token. SLSA provenance attestation is generated.
	5
	Blockchain Anchoring — artifact SHA-256 hash + provenance metadata written to blockchain smart contract. Immutable record created.
	6
	Policy Gate — OPA evaluates deployment readiness: is the artifact signed? Is its CVSS score below threshold? Is provenance verified?
	7
	Deployment — Kubernetes rolls out the artifact. Falco runtime agent starts monitoring syscalls, network connections, and file access.
	8
	Runtime Monitoring — any deviation from expected container behaviour (e.g., shell spawned inside container) triggers an immediate alert and optional automatic pod isolation.

PROJECT 03
VulnSight
AI-Powered Vulnerability Scanner
	

Overview & Strategic Context
VulnSight is a machine-learning-based code vulnerability scanner that trains on labelled vulnerability datasets and detects security flaws in source code automatically. It goes beyond pattern matching (like traditional SAST) by learning from real-world CVEs, bug bounty reports, and security research to identify novel vulnerability patterns.
The shift toward AI-native scanning is being driven by the explosion of AI-generated code in enterprise codebases. At the DevSecOps Summit 2025, industry leaders highlighted that LLM-generated code proliferates without adequate security review, creating unprecedented attack surfaces. Traditional rule-based scanners cannot keep pace — learned models that adapt to new patterns are the solution.


Problem Space
✔  Manual code review is slow — a senior reviewer analyses ~150 lines/hour at peak; large codebases are impossible to manually audit
	✔  Rule-based scanners generate excessive false positives — teams experience 'alert fatigue' and begin ignoring warnings
	✔  AI-generated code may contain subtle flaws not matching any existing rule pattern
	✔  Novel vulnerability classes emerge faster than rule sets can be updated
	✔  Context-blind analysis — tools that flag without understanding call graphs and data flows miss critical exploitable paths
	

How It Works — Scanning Pipeline
1
	Code Ingestion — source code is uploaded via API or connected to a Git repository. Supported: Python, JavaScript, Java, Go, Rust, C/C++.
	2
	Pre-processing — abstract syntax tree (AST) extraction, control-flow graph (CFG) construction, data-flow analysis.
	3
	Feature Engineering — numerical feature vectors derived from: function signatures, taint flows, API call sequences, cyclomatic complexity, dependency graph metrics.
	4
	ML Inference — trained ensemble model (Gradient Boosting + Graph Neural Network) predicts vulnerability class and severity for each code unit.
	5
	Reachability Analysis — exploitability is assessed: is the vulnerable function actually reachable from an attacker-controlled input?
	6
	Risk Scoring — each finding gets a CVSS-derived risk score (0–10) and a confidence rating (low/medium/high).
	7
	Remediation Suggestions — LLM-powered fix suggestions generated per finding, referencing CWE and OWASP Top 10 mitigations.
	8
	Report Generation — structured JSON/HTML report, dashboard visualisation, CI/CD pipeline integration.
	

Vulnerability Classes Detected
✔  Injection Flaws — SQL injection, OS command injection, LDAP injection, XPath injection
	✔  Insecure Deserialization — Java ObjectInputStream, Python pickle, PHP unserialize
	✔  Broken Authentication — hardcoded credentials, weak session management, JWT algorithm confusion
	✔  Cryptographic Failures — weak cipher selection, hardcoded secrets, improper key management
	✔  SSRF (Server-Side Request Forgery) — unvalidated URL inputs in HTTP client calls
	✔  Path Traversal — unvalidated file paths, directory listing exposure
	✔  Race Conditions — TOCTOU (time-of-check-time-of-use) flaws in multi-threaded code
	✔  Insecure Direct Object Reference (IDOR) — missing authorisation checks on resource access
	✔  Log Injection — unescaped user input in logging statements
	✔  Supply Chain: Dependency Confusion — package name squatting, typosquatting detection

CloudWeaver
GitOps Multi-Cloud Platform
	

Overview & Strategic Context
CloudWeaver is a GitOps-native platform for managing infrastructure and application deployments across multiple cloud providers from a single Git source of truth. By 2025, GitOps has matured from a Kubernetes-specific delivery method to a foundational operational standard, with ArgoCD used in nearly 60% of surveyed Kubernetes clusters per the 2025 CNCF end-user survey.
The platform engineering movement has positioned GitOps as the 'plumbing' of modern cloud-native delivery — ArgoCD or Flux are now day-one tools for platform teams. Akuity's KubeCon NA 2025 announcement extended GitOps to support not just Kubernetes but also VMs and serverless platforms, reflecting the industry's push toward hybrid multi-cloud GitOps.


Problem Space
✔  Configuration Drift — live cluster state diverges from declared state, causing silent failures
	✔  Manual Deployment Errors — ad-hoc kubectl or console changes bypass review and leave no audit trail
	✔  Inconsistent Environments — dev / staging / production diverge over time
	✔  Slow Recovery — manual processes mean hours to recover from failed deployments
	✔  Multi-Cloud Complexity — separate toolchains for AWS, GCP, Azure multiply operational overhead
	✔  Secrets Management — environment-specific credentials scattered across pipelines and manifests
	

How It Works — GitOps Loop
1
	Developer updates YAML manifests or Terraform HCL files in the Git repository via a pull request.
	2
	PR Review — peers review infrastructure changes; security policy checks run via OPA/Kyverno. Required approvals gate the merge.
	3
	Merge Triggers — CI pipeline runs Terraform plan, generating a change preview ("drift report") showing exactly what will be created, modified, or destroyed.
	4
	ArgoCD Reconciliation — the GitOps operator detects the Git commit and automatically applies the updated Kubernetes manifests to the target cluster.
	5
	Terraform Apply — infrastructure provisioning runs in an isolated runner; new cloud resources (VPCs, databases, load balancers) are created or updated.
	6
	Health Check — ArgoCD monitors resource health (pod readiness, service endpoints). Unhealthy resources trigger automatic rollback to the previous Git commit.
	7
	Observability — Prometheus scrapes cluster metrics; Grafana renders deployment timelines, resource utilisation, and error rates.
	8
	Drift Detection — ArgoCD continuously compares live state to Git state. Any out-of-band change is flagged immediately and can be auto-remediated.

PROJECT 05
ProofForge
Blockchain Supply Chain Attestation
	

Overview & Strategic Context
ProofForge provides cryptographic attestation of software artifacts using blockchain technology. Every build artifact is hashed and recorded on-chain, creating an immutable, verifiable chain of custody from source code commit to production deployment.
By 2025, software supply chain security has matured into a compliance requirement. The US Executive Order 14028 mandated SBOMs for federal software procurement. SLSA 1.0 was released as a vendor-neutral framework. Sigstore became the de-facto standard for keyless artifact signing, with GitHub Actions natively generating SLSA provenance via OIDC tokens. The SLSA framework's in-toto attestation format (ITE-6) defines a standard envelope used by cosign, SLSA, and blockchain tools alike.


Problem Space
✔  SolarWinds-class attacks — build systems compromised, malicious code injected into signed binaries
	✔  Log4Shell-class vulnerabilities — transitive dependencies introduce critical flaws invisible to developers
	✔  Dependency Confusion — attackers publish packages matching internal names to public registries
	✔  Unsigned Artifacts — no way to verify who built a binary and under what conditions
	✔  Audit Trail Gaps — after a breach, investigators cannot reconstruct the build provenance
	✔  Centralised Signing Failure — single HSM or signing server becomes a high-value attack target
	

How It Works — Attestation Flow
1
	Build Trigger — developer merges code; GitHub Actions CI run starts in an ephemeral, isolated environment.
	2
	Artifact Construction — Docker image built; SBOM generated (Syft → SPDX/CycloneDX format) listing every dependency.
	3
	Hash Computation — SHA-256 digest computed for the image manifest and each SBOM component.
	4
	Keyless Signing — cosign signs the image using GitHub Actions OIDC token via Sigstore Fulcio CA; signature logged to Rekor transparency log.
	5
	SLSA Provenance — signed in-toto attestation records: builder identity, source repo, git commit SHA, build parameters, environment variables.
	6
	Blockchain Anchoring — attestation hash + IPFS CID written to smart contract; block timestamp provides legal timestamping.
	7
	Registry Push — signed image + attestation pushed to OCI registry (GHCR, ECR, Artifact Registry).
	8
	Deployment Verification — Kyverno admission controller verifies: image is signed, provenance matches expected builder, SBOM exists, no critical CVEs in VEX block.
	9
	Audit Query — at any time, any stakeholder can query the smart contract to retrieve the full provenance chain for a given image digest.

Post-Quantum Blockchain Explorer
	

Overview & Strategic Context
QuantumLedger is a blockchain explorer enhanced with post-quantum cryptography (PQC), providing both a visualisation tool for blockchain data and a research platform for quantum-resistant cryptographic primitives.
In August 2024, NIST released its first three finalised PQC standards after an eight-year international evaluation of 82 algorithms from 25 countries. The three standards — ML-KEM (FIPS 203), ML-DSA (FIPS 204), and SLH-DSA (FIPS 205) — are now ready for immediate deployment. In March 2025, NIST selected HQC as a fifth algorithm (backup for ML-KEM based on different mathematical assumptions). NIST is urging all organisations to begin migration now, as the threat model includes 'harvest now, decrypt later' attacks where adversaries collect encrypted data today to decrypt once quantum computers are capable.


The Quantum Threat — Why It Matters Now
Harvest Now, Decrypt Later (HNDL): Adversaries are already collecting encrypted blockchain transactions today.
They will decrypt them retroactively once a Cryptographically Relevant Quantum Computer (CRQC) exists.
Experts estimate a CRQC capable of breaking RSA-2048 / ECDSA may emerge within 10–15 years.
Classical digital signature algorithms (ECDSA used in Bitcoin/Ethereum) will be broken by Shor's Algorithm.
NIST plans to deprecate classical public-key algorithms at the 112-bit security level by 2031.


PQC Algorithms Reference
Category
	Details
	ML-KEM (FIPS 203)
	Module-Lattice Key Encapsulation Mechanism. Primary algorithm for key exchange. Based on hardness of lattice problems. Replaces ECDH.
	ML-DSA (FIPS 204)
	Module-Lattice Digital Signature. Primary algorithm for signatures. Based on CRYSTALS-Dilithium. Replaces ECDSA.
	SLH-DSA (FIPS 205)
	Stateless Hash-Based Digital Signature. Based on SPHINCS+. Hash-based security, conservative design. Alternative signature scheme.
	FN-DSA (FIPS 206)
	Falcon lattice-based signature. Smaller signatures than ML-DSA but more complex implementation. Standard in development.
	HQC (March 2025)
	Hamming Quasi-Cyclic. Backup KEM for ML-KEM. Based on error-correcting codes (different math assumption). Standard expected 2027.
	

How It Works — Explorer Pipeline
1

	2
	Transaction Parsing — each transaction is decoded: sender, receiver, value, input data, logs, receipt.
	3
	PQC Signature Verification — if a transaction includes a PQC-signed payload (in calldata), the explorer verifies the ML-DSA or SLH-DSA signature.
	4
	ZK Proof Verification (optional) — Zero-Knowledge proof components (zk-SNARKs) are verified for privacy-preserving transactions.
	5
	Indexing — parsed data stored in PostgreSQL + Elasticsearch for fast query and search.
	6
	API Layer — REST + WebSocket API exposes block data, transaction data, PQC verification results.
	7
	Frontend Rendering — React dashboard renders block explorer, transaction graph, address analytics, and PQC verification status.
	8
	Cryptographic Benchmarking — built-in performance comparison between classical (ECDSA) and PQC (ML-DSA) operations.

PROJECT 07
NeuroOps
AI-Augmented DevSecOps Pipeline
	

Overview & Strategic Context
NeuroOps is an AI co-pilot for DevSecOps teams that automates code review, log analysis, threat summarisation, and operational recommendations. It embeds large language models (LLMs) and agentic AI into the day-to-day engineering workflow, reducing manual cognitive load while improving security outcomes.
The 2025 DevSecOps landscape is defined by the rise of agentic AI — autonomous agents that perform goal-driven security tasks with minimal human supervision. Unlike traditional automation (which runs fixed scripts), agentic AI understands context, reasons about risk, and adapts its actions. Platforms like Harness, GitHub (Copilot Autofix), and Snyk are embedding LLMs directly into remediation workflows. By late 2025 and into 2026, AI-driven automation is expected to fundamentally reshape how DevSecOps operates.

PROJECT 08
ChainGuard
Secure Supply Chain System
	

Overview & Strategic Context
ChainGuard is an end-to-end secure supply chain platform that tracks, verifies, and enforces integrity across the entire software development lifecycle — from source code to deployed artifact. It combines cryptographic verification, policy-as-code, and blockchain anchoring to deliver provenance you can prove.
By May 2025, supply chain security has become table stakes for federal, financial, health-tech, and enterprise SaaS organisations. Open standards have stabilised: SLSA 1.1 RC2 (April 2025), SPDX 3.0, and Sigstore's keyless signing ecosystem are now vendor-neutral baselines. The GUAC (Graph for Understanding Artifact Composition) project, contributed by Google, provides supply chain metadata aggregation at scale. The key insight from Faith Forge Labs' 2025 report: 'tooling is mature — SLSA Level 2 achievable in an afternoon; Level 3 in weeks.'


Problem Space
✔  Build System Compromise — attackers target CI/CD infrastructure to inject malicious code upstream of signing
	✔  Dependency Confusion — internal package names exploited by malicious public packages
	✔  Typosquatting — malicious packages with near-identical names to popular libraries
	✔  Unsigned Production Artifacts — no cryptographic guarantee of who built a binary
	✔  SBOM Gaps — components present in production but absent from declared inventory
	✔  Audit Evidence — post-breach forensics cannot reconstruct the full build provenance
	✔  Compliance Mandates — EO 14028, SSDF, and sector-specific regulations require SBOM + provenance
	

How It Works — Full Supply Chain Pipeline
1
	Source Verification — developer commit is signed with SSH key or GPG. Branch protection requires signed commits. Gittuf enforces repository security policies.
	2
	Dependency Scanning — every dependency updated via automated Renovate/Dependabot PRs. SCA scanner (Snyk/Trivy) checks CVEs; Capslock analyses capability risk.
	3
	Build Isolation — build runs in an ephemeral, hermetically sealed GitHub Actions runner. No network access to public internet; only allowlisted internal registries.
	4
	SBOM Generation — Syft generates a CycloneDX 1.5 SBOM listing every component with version, hash, and licence. SBOM is cryptographically signed.
	5
	Artifact Signing — cosign signs the output image/binary. Signature and certificate logged to Rekor (Sigstore transparency log). No private key leaves CI.
	6
	SLSA Provenance — GitHub Actions OIDC token used to generate signed in-toto provenance attestation. Specifies: repository, branch, commit SHA, workflow, runner environment.
	7
	GUAC Ingestion — SBOM + provenance + VEX documents fed into GUAC knowledge graph. Enables cross-artifact dependency analysis at organisation scale.
	8
	Blockchain Anchoring — attestation root hash written to smart contract on Hyperledger Besu or Ethereum. Provides legally defensible timestamped record.
	9
	Policy Enforcement — Kyverno ClusterPolicy in Kubernetes admission controller verifies: signed image, valid SBOM, provenance matches expected builder, no unresolved critical CVEs.
	10
	Runtime Monitoring — Falco monitors deployed containers; any violation (unexpected process, network connection) triggers alert and optional pod quarantine.
	11
	Audit Portal — stakeholders query the provenance graph: 'show me every image in production that contains log4j > 2.14'.
